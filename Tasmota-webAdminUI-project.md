# Экосистема веб-интерфейсов Tasmota для ESP32: исследование для проекта ASTRA

**На рынке не существует зрелого React-дашборда для Tasmota** — это ключевой вывод исследования. TasmoAdmin на PHP остаётся единственным активно поддерживаемым инструментом для управления множеством устройств, но он архитектурно устарел. Проект ASTRA имеет реальную возможность занять пустующую нишу: облачный React-дашборд с прямым подключением к локальным ESP32/Tasmota-устройствам. Технически это стало возможным благодаря Chrome 142+ Local Network Access (октябрь 2025), который снимает ограничения mixed content для локальных устройств. Оптимальная архитектура — **MQTT over WebSocket** как основной канал связи с гибридным локальным доступом через HTTP API.

## 1. Существующие проекты и расширения веб-интерфейса Tasmota

### TasmoAdmin — единственный живой «тяжеловес»

**TasmoAdmin** (~961 звёзд на GitHub, 1693 коммита) — доминирующий инструмент управления Tasmota-устройствами, официально рекомендованный в документации проекта. Написан на **PHP 8.1+** с Nginx, распространяется через Docker-контейнер и доступен как аддон Home Assistant. Ключевые возможности: автоматическое обнаружение устройств через сканирование сети, централизованное управление прошивками, мониторинг статуса. Подключается к устройствам через **HTTP API** (прямые HTTP-запросы к каждому устройству). Основное ограничение — PHP-стек требует серверной инфраструктуры, а HTTP-поллинг создаёт нагрузку на устройства при большом количестве девайсов.

**Tasmota Device Manager (TDM)** (~698 звёзд) — десктопное Python/PyQt5 приложение, работающее через MQTT. Поддерживает автодетект устройств, OTA-обновления, управление таймерами. Однако проект **фактически мёртв** с 2020 года: пользователи сообщают о частых крашах и низкой производительности. Существует форк `tdmgr` на PyPI, но с минимальной активностью.

**TasUI** (< 100 звёзд) — единственная попытка создать zero-install браузерный дашборд на vanilla JavaScript, доступный по адресу tasui.shantur.com. Работает через HTTP API с CORS. Проект **заброшен** с 2020–2021 года, а его функция обнаружения устройств ломает конфигурацию SetOption73 в Tasmota ≥8.3.

### Современные фреймворки: робкие первые шаги

Единственная попытка создать React-дашборд — проект **Scharxi/tasmo-admin** на Next.js 15 + React + PostgreSQL + Prisma ORM + shadcn/ui. Он находится в ранней стадии разработки (менее 10 звёзд), предлагает управление устройствами, мониторинг энергопотребления и автообнаружение, но **далёк от production-ready** состояния. Проект **aetheryx/tasmota-frontend** — минималистичный React-интерфейс для управления Tasmota-лампочками, фактически personal project.

Существует несколько лёгких решений: **saihgupr/Tasmota-UI** (vanilla HTML/JS, devices.json для конфигурации, HTTP API); **PHP-дашборд k0jdd** из Tasmota Discussion #18075 (ручная конфигурация через .ini файл); **Browser-MQTT-Dashboard** — браузерный JavaScript с MQTT-over-WebSocket для визуализации данных.

### Экосистема тачскринов и дисплейных интерфейсов

Самый успешный проект экосистемы — **NsPanel Lovelace UI** (~3000+ звёзд), реализующий Lovelace-подобный интерфейс на аппаратной панели Sonoff NSPanel. Использует Berry-скриптинг (Tasmota-драйвер) + прошивку Nextion (TFT) + Python AppDaemon бэкенд. Поддерживает cover, switch, sensor, light, lock, fan, timer, media player. Требует MQTT-брокер и AppDaemon-контейнер.

**HASPmota** — встроенный в ядро Tasmota32-LVGL модуль, совместимый с openHASP JSON Lines форматом. Обеспечивает богатую графику через LVGL на ESP32 с дисплеями, поддерживает TrueType-шрифты и Material Design Icons. Это для **физических экранов**, не для браузеров, но демонстрирует зрелость Berry-стека Tasmota.

### Berry-скриптинг: мощный, но недоиспользованный

Berry на ESP32 предоставляет полноценный фреймворк для расширения веб-интерфейса через модуль `webserver`. Хуки `web_add_main_button()`, `web_add_config_button()`, `web_sensor()` позволяют добавлять произвольный HTML на страницы Tasmota. Функция `webserver.on(uri, callback, method)` регистрирует пользовательские HTTP-эндпоинты, а `webserver.content_response(json)` отдаёт JSON-ответы — фактически позволяя создать **кастомный REST API** на устройстве. Berry-скрипты хранятся в файловой системе LittleFS и автозагружаются через `autoexec.be`.

### Мониторинг через Grafana-стек

Для продвинутого мониторинга энергопотребления пользователи строят стек **Mosquitto MQTT → Python-скрипт → InfluxDB → Grafana** (проект tasmota-plug-monitoring на GitHub). Это единственный способ получить историческую аналитику без Home Assistant, но требует настройки 4 отдельных сервисов.

## 2. Анализ функциональности и потребностей пользователей

### Что предлагает стандартный веб-интерфейс Tasmota и почему он намеренно ограничен

Стандартный WebUI Tasmota — это **не приложение**, а набор HTML-шаблонов, вкомпилированных в прошивку как `PROGMEM`-константы в файле `xdrv_01_9_webserver.ino`. HTML генерируется на стороне ESP через printf-подобное форматирование и отправляется chunked HTTP-ответами. JavaScript минимален — в основном inline-функции для AJAX-подобных взаимодействий и автообновление страницы каждые **2345 мс**. Нет внешних зависимостей, CDN-ресурсов или фреймворков.

Интерфейс предоставляет: главную страницу (кнопки реле, показания сенсоров, управление светом), конфигурацию модуля/WiFi/MQTT/логирования, информационную страницу (версия прошивки, сетевые параметры, статистика чипа), обновление прошивки по OTA, консоль с потоком логов, конфигуратор **16 таймеров**, и на ESP32 — Berry-консоль с файловым менеджером.

Мейнтейнеры проекта **открыто позиционируют** WebUI как admin/test инструмент. Цитата sfromis (мейнтейнер): *«The web UI is really meant for admin/test, not for making a user interface for daily operation.»* Цитата Theo Arends (создатель Tasmota): *«Just find one who will want to rewrite the current GUI while keeping the code footprint low»* — в ответ на запрос улучшения UI для многоканальных реле.

### Десять главных болей пользователей

Анализ GitHub Discussions, Reddit r/tasmota и Home Assistant Community выявил устойчивый паттерн неудовлетворённого спроса:

1. **Невозможность настроить отображение элементов** — нельзя скрыть конкретные реле, переупорядочить элементы или добавить виджеты (Discussion #22650)
2. **Отсутствие мультиустройственного управления** — каждое устройство в отдельной вкладке; три стронних проекта (TasUI, TDM, TasmoAdmin) были созданы именно для решения этой проблемы
3. **Нет исторических графиков** — энергомониторинг показывает только текущие мгновенные значения; для графиков нужен стек Grafana+InfluxDB
4. **Правила (Rules) — только текстовый ввод** через консоль; нет визуального редактора, отладка сложна
5. **UI ломается при сложных конфигурациях** — кнопки неправильно выравниваются при >4–8 реле (Discussion #11546, подтверждено с MCP23017-расширителями)
6. **Нет разграничения доступа** — «admin или ничего», нельзя дать домочадцам простой интерфейс с кнопками без доступа к настройкам (Discussion #14987)
7. **Нет push-обновлений** — HTTP-поллинг каждые ~2.3 секунды вместо WebSocket
8. **Ограниченное расписание** — только 16 таймеров с базовым интерфейсом, нет календарных представлений
9. **Нет HTTPS** — весь трафик нешифрован, пароль WebPassword — «very thin layer of protection»
10. **Базовая мобильная адаптивность** — viewport meta tag есть, но интерфейс не оптимизирован под тач

### Сравнение с Home Assistant Lovelace: что заставляет ставить HA

Home Assistant Lovelace предоставляет всё, чего не хватает Tasmota WebUI, и именно это заставляет пользователей разворачивать полноценный HA-сервер только ради дашборда:

**Мультиустройственный дашборд** — неограниченное количество устройств на одном экране с drag-and-drop расположением карточек. **30+ встроенных типов карточек**: tile, gauge, history graph, energy, thermostat, media, map. **Экосистема кастомных карточек** через HACS — тысячи сторонних решений (Mushroom Cards, Bubble Card, Mini Graph Card). **Энергетический дашборд** — отслеживание потребления, солнечной генерации, стоимости с долгосрочной статистикой. **Визуальный редактор автоматизаций** с trigger → condition → action flow. **WebSocket push** — мгновенное обновление при изменении состояния, без поллинга. **Многопользовательская система** с ролями и правами. **Нативные мобильные приложения** для iOS и Android с push-уведомлениями.

Самые популярные Lovelace-карточки для Tasmota-устройств: **Tile card** (компактное управление), **History graph** (графики сенсоров во времени), **Energy card** (мониторинг энергопотребления), **Gauge card** (круговые индикаторы мощности/температуры), **Mushroom Cards** и **Bubble Card** из HACS.

### Фундаментальное противоречие

Философия проекта Tasmota: WebUI — это admin-инструмент, для дашбордов используйте HA/Domoticz/Node-RED. Но **значительная часть пользователей хочет standalone-операцию** без сложности разворачивания отдельного сервера умного дома. Именно этот разрыв между потребностью и предложением создаёт возможность для проекта ASTRA.

## 3. Архитектурная концепция облачного React-дашборда

### Коммуникационные интерфейсы Tasmota: что доступно

**HTTP API** — основной интерфейс: эндпоинт `/cm?cmnd=<COMMAND>` принимает GET/POST, возвращает JSON. Команда `Status 0` возвращает полный статус устройства. `Backlog` позволяет отправить до 30 команд за один запрос. Аутентификация через `user=admin&password=<WebPassword>`. Критический момент: **CORS отключён по умолчанию** в прекомпилированных прошивках (PR #12827 сделал это опциональным). Для включения нужна кастомная сборка с `#define CORS_DOMAIN "*"` или `SetOption121 1` в поддерживающих сборках.

**WebSocket — НЕ поддерживается нативно.** GitHub Issue #3368 подтверждает: попытка WebSocket-подключения к `/cm` возвращает HTTP 200 вместо 101 Upgrade. Мейнтейнеры явно заявили, что WebSocket не будет добавлен. Консоль Tasmota WebUI работает через **HTTP-поллинг** (XMLHttpRequest к `/ay` и `/cs` каждые ~2 секунды), а не через WebSocket.

**MQTT** — рекомендованный протокол для реального времени. Трёхпрефиксная структура топиков: `cmnd/<topic>/` (команды → устройству), `stat/<topic>/` (ответы ← от устройства), `tele/<topic>/` (телеметрия ← периодически, по умолчанию каждые 300 секунд). LWT-топик `tele/<topic>/LWT` публикует «Online»/«Offline» (retained). Поддержка Home Assistant Auto Discovery через `tasmota/discovery/#`.

**Berry Custom API (только ESP32)** — через `webserver.on(uri, callback)` можно зарегистрировать произвольные эндпоинты, включая JSON API. GPIO Viewer использует **Server-Sent Events (SSE)** для стриминга данных, что демонстрирует возможность push-обновлений на ESP32, хотя это не документированная публичная функция.

**Обнаружение устройств**: mDNS **отключён по умолчанию** из-за багов в Arduino-реализациях. Включается через `SetOption55 1` + кастомная сборка. Самый надёжный метод — MQTT-based discovery: подписка на `tele/+/LWT` для обнаружения, затем `Status 0` для получения деталей. Альтернатива — сканирование IP-диапазона с проверкой HTTP-ответа.

### Проблема Cloud → Local: как её решают другие

**Shelly** использует двухрежимную архитектуру: локальный HTTP REST API + CoIoT для multicast-обновлений + опциональное облако через home.shelly.cloud. Приложение Shelly Smart Control имеет переключатель «Local Wi-Fi Network Control» — при включении общается с устройствами напрямую, минуя облако. CORS можно включить в настройках устройства. Gen2+ устройства поддерживают JSON-RPC через WebSocket.

**ESPHome** — чисто локальная архитектура. Дашборд — это dev-инструмент на Tornado/Python (порт 6052). Устройства общаются с HA напрямую через TCP по собственному протоколу ESPHome API. Каждое устройство может опционально запустить web_server-компонент с SPA-дашбордом, устанавливаемым как PWA.

**Sonoff/eWeLink** — гибридная модель «SonoffLAN»: начальная загрузка списка устройств с облака, затем предпочтение локальных подключений, откат на облако при недоступности. Обнаружение через mDNS (`_ewelink._tcp`). Этот паттерн **«cloud-first, local-opportunistic»** — наиболее практичный для ASTRA.

### Chrome Local Network Access — переломный момент

**Chrome 142+ (октябрь 2025)** запустил Local Network Access (LNA) — замену нереализуемого подхода с CORS-preflight от IoT-устройств. Теперь при попытке HTTPS-сайта обратиться к локальным адресам Chrome показывает разрешение: *«[Сайт] wants to look for and connect to any device on your local network.»* После подтверждения **mixed content ограничения снимаются** для известных локальных целей: приватных IP-адресов (192.168.x.x), .local-доменов и fetch-запросов с `targetAddressSpace: "local"`. Это **критически важно** для ASTRA — облачный HTTPS-дашборд теперь может напрямую обращаться к HTTP API Tasmota-устройств без прокси.

Ограничение: Service Workers требуют, чтобы основная страница уже получила LNA-разрешение, прежде чем они смогут делать локальные запросы.

### Рекомендуемая архитектура: «Cloud-First, Local-Opportunistic»

```
┌─────────────────────────────────────────────────┐
│           React PWA (Vercel/Netlify, HTTPS)      │
│  ┌──────────┐  ┌────────────┐  ┌──────────────┐ │
│  │ Zustand   │  │ MQTT.js    │  │ Direct HTTP  │ │
│  │ Device    │  │ WebSocket  │  │ via Chrome   │ │
│  │ State     │  │ → Broker   │  │ LNA          │ │
│  └──────────┘  └──────┬─────┘  └──────┬───────┘ │
│       Service Worker: кэш shell + last-known     │
└─────────────────┬──────────────────────┬─────────┘
                  │ wss://               │ http://192.168.x.x
          ┌───────▼──────────┐   ┌───────▼──────────────┐
          │ MQTT Broker      │   │ ESP32/Tasmota         │
          │ (Mosquitto/EMQX) │   │ HTTP API + Berry API  │
          │ ws:9001/wss:9002 │   │ MQTT → Broker         │
          └───────┬──────────┘   └───────────────────────┘
                  │ mqtt:1883
          ┌───────▼──────────────────┐
          │ ESP32/Tasmota Devices    │
          │ cmnd/stat/tele topics    │
          └──────────────────────────┘
```

**Два параллельных канала связи:**

1. **MQTT over WebSocket** (основной) — браузер подключается к MQTT-брокеру через `wss://`, подписывается на `stat/<topic>/+` и `tele/<topic>/+`, публикует команды в `cmnd/<topic>/`. Это обходит все CORS-проблемы, обеспечивает bidirectional real-time связь, поддерживает QoS, retained messages и LWT для определения offline-устройств. Работает через NAT и файрволы.

2. **Direct HTTP** (дополнительный, через Chrome LNA) — для операций, не покрываемых MQTT: загрузка Berry-скриптов, OTA-обновление прошивки, получение полного Status 0, конфигурация. Запросы идут напрямую на `http://<device-ip>/cm?cmnd=...`.

**Обнаружение устройств**: гибридная схема — (a) загрузка списка устройств из облачного профиля пользователя, (b) MQTT-discovery через подписку на `tele/+/LWT`, (c) опциональное сканирование IP-диапазона через прямые HTTP-запросы.

### Паттерны реального времени: WebSocket vs MQTT vs HTTP polling

| Протокол | Задержка (LAN) | Направление | Масштабируемость | Применение для ASTRA |
|----------|---------------|-------------|-------------------|---------------------|
| MQTT over WS | 100–500 мс | Bidirectional | Отличная | Основной канал: состояние, телеметрия, команды |
| Direct HTTP (LNA) | 50–100 мс | Request/Response | Средняя | Конфигурация, OTA, Berry-скрипты |
| HTTP Polling | ~500 мс avg | Request/Response | Плохая | Не рекомендуется |
| SSE (Berry) | <10 мс | Server → Client | Ограничена | Возможна для кастомных ASTRA-эндпоинтов |

### Модульная архитектура виджетов по модели Lovelace

Lovelace Home Assistant построен на **Web Components** (LitElement). Каждая карточка реализует минимальный контракт: `setConfig(config)`, сеттер `hass`, `getCardSize()`, `getGridOptions()`. Конфигурация в YAML/JSON, кастомные карточки регистрируются через `customElements.define()` и распространяются через HACS.

Для React-дашборда ASTRA рекомендуется адаптированный подход:

**Widget Interface (TypeScript):**
```typescript
interface AstraWidget {
  id: string;
  name: string;
  version: string;
  getDefaultConfig(): WidgetConfig;
  getConfigSchema(): JSONSchema;  // → авто-генерация формы настроек
  render(props: { devices: DeviceState[], config: WidgetConfig }): ReactNode;
  onDeviceUpdate?(deviceId: string, data: any): void;
  destroy?(): void;
}
```

**Ключевые архитектурные решения:**

- **react-grid-layout v2** — движок разметки с drag-and-drop, resize, responsive breakpoints, сериализуемым layout. 2M+ weekly downloads, полная TypeScript-поддержка
- **Zustand** (~1KB) для device/UI state — fine-grained subscriptions, компоненты ре-рендерятся только при изменении их конкретного устройства. Persist middleware для IndexedDB/localStorage
- **TanStack Query** для server state — кэширование конфигураций, исторических данных, пользовательских настроек
- **JSON Schema → авто-генерация форм** для настроек виджетов (паттерн ThingsBoard — 600+ виджетов используют эту модель)
- **Dynamic import()** для ленивой загрузки виджетов, **Module Federation** для сторонних плагинов в будущем
- **Widget Registry** — центральный каталог виджетов с метаданными для UI-пикера

**Паттерны обработки реального времени:**
- **Буферизация MQTT-сообщений** — `BUFFER_INTERVAL = 1000ms`, UI обновляется максимум 1 раз в секунду для предотвращения render storms
- **Батчинг** — сбор событий за 100–200 мс, затем однократное обновление state
- **Ring Buffer** — кольцевой буфер для метрик с rolling averages
- **Stale indicators** — маркеры устаревших данных при потере связи

### PWA-стратегия для offline и мобильных устройств

React-дашборд ASTRA должен быть PWA с чёткой стратегией кэширования:

- **App shell** — Cache-First (мгновенная загрузка всегда)
- **Список устройств/конфигурация** — Stale-While-Revalidate (показать кэш немедленно, обновить в фоне)
- **Данные сенсоров** — Network-Only с IndexedDB fallback (отображать last-known values с таймстемпами при offline)
- **Исторические графики** — Cache с time-based expiry
- **Background Sync** — очередь команд при offline, воспроизведение при восстановлении связи
- **Push Notifications** — оповещения о событиях устройств (пороги сенсоров, offline-статус)

### Дорожная карта поэтапной реализации

**Фаза 1 — MVP (4–6 недель):** MQTT-подключение через WebSocket, отображение статуса устройств (online/offline/power/sensor), кнопки управления реле, ручное добавление устройств по MQTT-топику. Минимальный набор виджетов: relay toggle, sensor display, device info card.

**Фаза 2 — Device Management (6–8 недель):** Direct HTTP доступ через Chrome LNA для конфигурации, автообнаружение устройств через MQTT LWT, настройка WiFi/MQTT/модуля через дашборд, OTA-обновление прошивки, Berry-консоль.

**Фаза 3 — Advanced Dashboard (8–12 недель):** react-grid-layout с drag-and-drop, расширенный набор виджетов (gauge, history graph, energy dashboard), JSON Schema конфигуратор виджетов, визуальный редактор таймеров, PWA с offline-поддержкой.

**Фаза 4 — Automations & Extensibility (12+ недель):** Визуальный редактор Rules (node-based или trigger→condition→action flow), сценарии с межустройственным взаимодействием, Module Federation для сторонних виджетов, многопользовательская система с ролями.

## Ключевые выводы и стратегические рекомендации

Рынок дашбордов для Tasmota **пуст** — ни один проект не сочетает современный фронтенд, реальное время и multi-device управление. TasmoAdmin устарел архитектурно, TasUI и TDM мертвы, а новые React-попытки находятся в зачаточном состоянии. Проект ASTRA имеет возможность стать первым production-ready решением в этой нише.

**MQTT over WebSocket — безальтернативный выбор** для основного канала связи. Tasmota не поддерживает WebSocket нативно и не планирует, CORS отключён по умолчанию — MQTT-брокер решает обе проблемы одновременно, обеспечивая bidirectional push-обновления. Chrome 142+ LNA открывает дополнительный прямой канал для конфигурационных операций.

Архитектура Lovelace Home Assistant — лучший прототип для системы виджетов, но с критической адаптацией: **контракт виджета должен быть проще** (3–4 метода), конфигурация через JSON Schema с авто-генерацией UI, а не YAML. Zustand вместо глобального `hass`-объекта обеспечит fine-grained re-rendering при высокочастотных обновлениях от множества устройств.

Стратегическое преимущество ASTRA: **Berry-скриптинг на ESP32** позволяет добавить на устройства кастомные API-эндпоинты, SSE-стриминг и CORS-заголовки, адаптируя прошивку под нужды дашборда — чего не может сделать ни один внешний проект, работающий только с vanilla Tasmota.
